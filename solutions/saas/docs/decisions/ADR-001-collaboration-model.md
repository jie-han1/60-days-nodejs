# ADR-001：协作模型 — 工作区（团队）+ 成员身份，不做多租户隔离

- **状态**：已采纳（Day 46）
- **影响**：所有业务表、所有授权判断

## 背景（Context）

任务管理工具要支持「几个人一起管任务」。最朴素的问题是：**数据归谁所有、谁能看见**。摆在面前有两种思路，复杂度差一个量级：

- **A. 多租户隔离**：把每个工作区当成「互不信任的客户」，数据层做硬隔离（每行冗余 `orgId`、Row-Level Security、跨工作区绝对不可见）。这是卖给企业、合规要求高的 SaaS 的做法。
- **B. 协作分组**：把工作区当成「一个团队」，数据归属走「项目属于工作区，能看见 = 你是成员」，应用层鉴权把好关即可，数据层不做隔离机器。

## 决策（Decision）

**选 B：工作区（Workspace）是协作单位，数据归属走「所有权 + 成员身份」，数据层不做多租户隔离。**

具体说：

- 每个项目属于一个工作区，每个任务属于一个项目。归属链 `Task → Project → Workspace`。
- 能看见/操作一个资源 = **你是它所在工作区的成员，且角色够**。授权判断顺着归属链回溯到工作区，查 `Membership(userId, workspaceId)`。
- **不**在每行冗余工作区 id，**不**上 RLS。一个查询能不能返回某行，靠应用层「先验成员身份、再按工作区范围查」保证。

理由，从「这个产品是什么」倒推：

- 它是**一个团队内部协作的工具**，不是「同一套软件服务多个互不信任客户」的企业 SaaS。工作区里的成员**本来就互相可见**——不存在「A 工作区的人不该看见 B 工作区」这种跨租户威胁模型，只有「工作区外的人不该进来」这一层，应用层鉴权足够。
- 多租户隔离那套（orgId 冗余 + RLS + 跨租户审计 + 每租户备份策略）的复杂度，只有在「客户之间互不信任且数据泄漏是合规事故」时才回本。对这个学习项目是纯负担。
- 应用层鉴权的语义清晰、好测：「能不能做 = 是不是成员 + 角色够不够」。把这道关写对、测好，比在数据层加一层 RLS 更直接。

## 后果（Consequences）

**好的：**
- 数据模型干净：没有 `workspaceId` 满天飞，归属关系就是 `Task → Project → Workspace` 一条链。
- 开发简单：写查询不用每条都带 `WHERE workspace_id = ...`，授权集中在「进 handler 前验一次成员身份」。
- 心智负担低：理解「谁能看见什么」只需要懂一条归属链，不需要懂 RLS 策略。

**坏的 / 要认的代价：**
- **隔离全靠应用层自觉**。万一某条查询忘了先验成员身份、直接按 id 查了别人的任务，数据层没有 RLS 兜底——会泄漏。缓解：授权集中在一个中间件/guard 里做，不分散到每个 handler；配测试覆盖「非成员访问 → 403」。
- **不适合「互不信任客户」的场景**。哪天真要卖给「数据绝不能互相看见」的企业客户，这个模型不够，得往 A（多租户隔离）迁。但那是另一个产品阶段的事。
- **跨工作区的全局查询容易写**（这既是优点也是隐患）：没有隔离约束，`SELECT * FROM tasks` 会跨所有工作区——后台运营脚本方便，但写错了也容易把所有人的数据搅在一起。运营类查询要显式限定范围。

## 什么时候推翻这个决策

明确的升级信号：**出现「工作区 A 的数据绝不能被工作区 B 的任何人看见，包括平台运营」这种合规要求**（典型是金融/医疗客户、或签了 DPA 的企业）。那时再上多租户隔离——因为归属链已经是 `Task → Project → Workspace`，迁移到「每行带 workspaceId + RLS」是机械工作，不是重写应用。

> 这个决策是「按真实威胁模型选复杂度」的典型：威胁模型里没有「跨租户互不信任」，就不付多租户的税。把多租户留给真有多租户需求的产品阶段。
