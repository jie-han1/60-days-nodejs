# ADR-004：软删除 — 内容实体可恢复，账号关系硬删

- **状态**：已采纳（Day 46）
- **影响**：Task / Comment / Project，以及所有列表查询

## 背景（Context）

「删除」在一个协作工具里是高危操作：用户误删了一张卡、清空了一个项目，或者老板要求「找回上周删掉的任务」。如果 `DELETE` 是物理删除（`DELETE FROM tasks WHERE id=...`），数据就没了，只能靠数据库备份整库恢复（PITR，重且慢）。

但**全部软删除**也不对：账号、成员关系这类「撤了就该真撤」的，留着一行僵尸数据反而碍事（用户退出了组织，成员关系就该消失，不能软删残留还占着座位）。

## 决策（Decision）

**按「可恢复性有没有业务价值」分两类，区别对待：**

- **内容实体（Task / Comment / Project）→ 软删除**。加 `deletedAt: DateTime?`，删除 = `UPDATE ... SET deleted_at = now()`。30 天后由定时任务物理清理（或留着、给企业版做「回收站」）。schema 里这三张表都带 `deletedAt`。
- **账号与关系（User / Membership / Invitation）→ 硬删除 + 级联策略**。用户注销账号 = 物理删（合规要求「彻底删除我的数据」）；成员被移除 = 物理删 Membership 行。它们的「删除」没有恢复的语义。

## 后果（Consequences）

**软删除带来的、必须配套做的几件事（不做就是坑）：**

- **所有列表查询默认带 `WHERE deleted_at IS NULL`**。忘了带，回收站里的卡又出现在看板上。靠在仓储层统一拦截（不是每个 handler 自己加），或靠 Prisma 的 middleware / RLS 策略兜底。
- **唯一约束要兼容软删行**。比如 `@@unique([projectId, number])`（任务编号）：如果软删的行还占着 `ENG-42`，新建的任务就不能再用 42。两种解法：要么编号只用递增不回收（推荐，编号本就该单调），要么唯一约束改成 `(projectId, number, deleted_at)` 让软删行让出位置——这里选前者，编号不回收更符合直觉。
- **外键引用要定好 `onDelete`**。任务软删了，它的评论怎么办？schema 里 Comment 随 Task `onDelete: Cascade`（任务真删时评论级联删；软删则评论跟着保留）。assignee 注销账号时 `onDelete: SetNull`（任务留着，负责人置空，不级联删任务）。
- **「彻底删除」是独立动作**。软删是「放回收站」；定时任务或管理员才能「清空回收站」（物理 DELETE）。两个端点，别混。

**好的：**
- 误删可恢复，协作工具的「安全感」到位。
- `deletedAt` 记录了删除时间，排查「这卡什么时候没的」有据可查（更完整的「谁删的」要等加审计日志）。

**坏的 / 要认的代价：**
- **表会慢慢积累软删行**，占空间、拖慢扫描型查询。靠定期物理清理抵消。
- **每个列表查询多一个谓词**，心智负担 + 容易漏。靠仓储层统一拦截降低出错率，但风险面真实存在。
- **GDPR「彻底删除」要二次操作**（软删 → 物理删），不能一步到位。

## 为什么不全部硬删 + 靠备份恢复

因为「找回某一张特定的卡」和「整库恢复到某个时间点」是两种粒度。PITR 是把整个库回滚，会丢掉恢复点之后**所有人**的新数据——为了找回一张误删的卡把全公司的活儿回滚，不可接受。软删除把「找回」变成一条 `UPDATE deleted_at = NULL` 的廉价操作，粒度对了。这就是软删除存在的根本理由：**它卖的是「单条可恢复」的粒度，备份给不了这个粒度**。
