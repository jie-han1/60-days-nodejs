# ADR-002：同步优先，实时协作后置

- **状态**：已采纳（Day 46）
- **影响**：Day 47-60 的功能排期

## 背景（Context）

任务管理工具的「酷」版本（Linear、Notion）都有实时协作：两个人同时打开一张卡，A 改了标题，B 的屏幕上瞬间变；A 的光标在描述里移动，B 能看到。这套体验背后是 CRDT（无冲突复制数据类型）或 OT（操作变换）+ WebSocket + 状态广播，工程复杂度极高。

问题是：**Day 46 在画一个 MVP 的架构，不是在画一个 Linear 的架构**。如果一开始就奔着实时协作去，会把一个「还没人用的产品」拖进最难、最贵、且最依赖真实并发场景才能调对的技术深水区。

## 决策（Decision）

**MVP 全程同步（请求-响应），实时协作明确推到产品验证之后。** 具体分三档：

1. **第一档（Day 47-50 做的）：纯同步 REST/tRPC**。建卡、改字段、拖看板都是「请求 → 落库 → 返回新状态」。并发改同一张卡靠乐观锁（`If-Match`/`version`，见 api-design §6）兜底——两人同时改，第二个收 409，前端刷新重试。够用，且正确。
2. **第二档（产品有人用了再做）：轻量「新鲜感」**。两个低成本的近实时手段，先于完整实时协作：
   - **轮询**：列表页每 30s 重新拉一次。粗暴，但对「知道别人建了新卡」够用，零额外基建。
   - **SSE / WebSocket 广播**：服务端在数据变更时（建卡/改状态）往一个组织频道推「这条变了，去拉一下」的通知，前端收到后**重新拉数据**（不是应用增量 patch）。这是「失效通知」，不是「CRDT 同步」——复杂度低一个量级，体验已经接近实时。
3. **第三档（真有强协作需求才做）：完整实时协作**。多人光标、无冲突合并、离线编辑。这时才上 CRDT（Yjs/Automerge）或自研 OT。这是独立的大主题，不在本弧线。

## 后果（Consequences）

**好的：**
- Day 47 起的开发速度不被实时基建拖住，能尽快拿到「能用」的产品给真人用。
- 并发改动有明确、可测的语义（乐观锁 409），不用一上来就解 CRDT 的合并难题。
- 第二档的「SSE 失效通知」是渐进的：它的基建（频道广播）和第三档共享一部分，不算白做。

**坏的 / 要认的代价：**
- **多人高频协作的场景体验打折**。两个人同时编辑描述的长文本，第二档只能做到「保存时 409、丢掉一方」，不如 CRDT 的自动合并。这是为速度付的体验税。
- **第二档的「重新拉全量」比 CRDT 的增量贵**。一次变更触发整列表重拉，高频变更时流量放大。缓解：失效通知带「精确到哪条 task 变了」，前端只重拉那一条。

## 什么时候升级

判断信号很具体：**当用户开始抱怨「我看不到同事刚改的」或「我们改冲突了」频次变高**，就是从第二档升第三档的时机。在那之前，乐观锁 + 失效通知的性价比远高于 CRDT。

> 这个决策是「YAGNI（你不会需要它）」在架构层的典型应用：**把昂贵的通用方案，推迟到它真正被需要的那一刻**。多数 SaaS 死于过早建设，而非功能不足。
