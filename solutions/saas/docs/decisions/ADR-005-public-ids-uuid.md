# ADR-005：公开 ID 用 UUID，对外编号用项目内自增

- **状态**：已采纳（Day 46）
- **影响**：所有主键、URL、对外展示的编号

## 背景（Context）

ID 的选择看似细节，实则一次性、且影响所有表和 URL。两个诉求打架：

1. **不可枚举、不泄漏规模**。用自增整数（1, 2, 3...）当公开 id，攻击者枚举 `/tasks/1`、`/tasks/2` 就能扒光，还能从最大 id 推出「这项目有多少任务」。商业上，团队也不想让人一眼看出自己是第几个用户。
2. **对人友好、可口头引用**。「去看下 ENG-42 那张卡」——协作里，任务需要一个短、好念、好记的编号。

这两个诉求，一个 id 满足不了。

## 决策（Decision）

**两层 id，各司其职：**

- **主键 / 内部关联 → UUID**。schema 里所有 `id` 都是 `@default(uuid()) @db.Uuid`。不可枚举、不泄漏规模、全局唯一（合并数据/分库都不撞）、分片友好。
- **对外展示 / URL / 引用 → 项目内自增编号**（`ENG-42`）。`number` 是项目内单调递增的整数，前缀是项目 key。人念、写在 commit message 里、贴在聊天里都方便。

对外暴露的是编号（URL: `/workspaces/acme/projects/ENG/tasks/42`），UUID 留在内部做关联，不进对外 URL。

## 后果（Consequences）

**好的：**
- 攻击者无法枚举公开 URL（UUID 不可猜），规模信息不泄漏。
- 团队协作有顺口的编号（ENG-42），比念一串 UUID 体验好两个量级。
- UUID 全局唯一，未来要做「分库分表」或「合并多个库」时主键不撞——这是为「长大」留的口。

**坏的 / 要认的代价：**
- **UUID 占空间、索引比整数大**。`uuid` 16 字节 vs `int4` 4 字节，索引也更大。对任务管理这个量级完全可接受；超高写入场景才需要权衡。
- **编号的并发生成要小心**（见下）：不能裸 `MAX+1`，得用计数器行锁。
- **两套 id 要在文档/接口里讲清谁对外**。前端 URL 用编号，内部关联（外键）用 UUID。约定要一致，否则混乱。

## 附：任务编号（"ENG-42"）的并发安全

「项目内自增编号」藏着个并发问题。朴素做法 `SELECT MAX(number)+1` 在两个并发建卡时会拿到同一个号。正解两种：

1. **每项目一行计数器 + 行锁**：`project_task_counters(project_id, next_number)`，建卡时 `UPDATE ... SET next_number = next_number + 1 RETURNING next_number`（行锁串行化同项目的并发建卡），拿到号再插任务。同事务内完成，号连续。
2. **PG 每项目一个 sequence**：`CREATE SEQUENCE task_seq_eng`，`nextval()` 取号。sequence 的 `nextval` 是原子的，但 sequence 不参与事务回滚（建卡失败会「浪费」一个号，号不连续）——对「人能引用的编号」通常可接受。

Day 47 落地时选 1（计数器行，事务一致、号连续）。两种都成立，记在这里免得到时候重新想。

## 为什么不上 UUID v7 / 不讨论「UUID v4 vs v7」太多

schema 里 `@default(uuid())` 生成的是 **v4（随机）**。v7（时间有序）能让 B-tree 索引插入更顺（随机 v4 插入会频繁分裂索引页），是「写入量大到索引成为瓶颈」时的优化。Prisma 原生 `uuid()` 目前是 v4；要上 v7 得用 `dbgenerated` 或应用层生成（如 `uuidv7` 库）。**对一个 MVP，v4 完全够，v7 是过早优化**——记在这里，等真写入瓶颈了再换，因为那时候的迁移（改默认值生成方式）比改 ID 类型便宜得多。

> 判断标准：**v4 是「现在够用、将来好换」的默认；只有当「索引插入抖动」真成为可观测的问题时，才值得上 v7。** 这条和 ADR-002 的 YAGNI 是一个调子——别为没出现的问题付现在的复杂度。
